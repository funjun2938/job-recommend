조원 누구나 이 한 장으로 우리 DB가 어떻게 생겼고, 데이터가 어디서 와서 어떻게 추천이 되는지 알 수 있게 정리했어요.

한눈에 보기 데이터 2축 전체 ERD 테이블 14개 관계 설명 데이터 흐름 추천 점수
Overview

핵심은 "데이터를 만드는" 구조예요

단순히 추천만 하는 게 아니라, 사용자가 정보를 넣어야 추천을 받는 가치 교환으로 외부에 없는 데이터를 모읍니다. 데이터는 크게 두 군데에서 들어오고, 마지막에 하나의 추천 점수로 합쳐집니다.

📌 딱 3줄 요약
① 사용자가 직접 입력하는 서베이 데이터 (실연봉·퇴사이유·회사 평판)
② 명함첩에서 가져오는 네트워크 데이터 (인맥 회사 맵 · 이직 이력)
③ 둘을 합쳐 recommendation_scores 한 테이블에 최종 추천 점수로 저장
서베이 축 명함첩 축 추천 점수 유틸/지원
Two Data Axes

데이터는 두 군데에서 들어와요

왜 두 개냐? 설문만으로는 "내 인맥"이나 "비슷한 사람들의 실제 이직 경로"를 알 수 없기 때문이에요. (교수님 피드백 반영)

축 ① 서베이

사용자가 직접 입력

외부에 공개되지 않는 회사 내부 정보를 본인이 입력합니다.

  • ▪company_insights — 실연봉·퇴사이유·장단점·NPS (핵심 자산)
  • ▪company_reputation — 회사별 평판 자동 집계
  • ▪salary_benchmarks — 직군·규모별 연봉 분포
축 ② 명함첩 (외부 연동)

리멤버 류에서 가져옴

명함첩을 연동하면 두 가지 새 데이터가 생깁니다.

  • ▪명함첩 = 인맥 회사 맵 → company_network_edges (레퍼럴 유망 회사)
  • ▪명함 교체 = 그 사람의 이직 → contact_transitions + career_paths
  • ▪나와 비슷한 커리어를 탄 사람의 다음 회사 추천
Full ERD

전체 관계도

화살표는 테이블 간 연결을 뜻해요. 색이 다른 그룹(서베이/명함첩)이 마지막에 추천 점수로 모입니다.

erDiagram
    user_profiles ||--o{ namecards : "명함첩 소유 (1:N)"
    user_profiles ||--o{ recommendation_scores : "추천 결과 (1:N)"

    namecards }o--|| contact_transitions : "명함 교체 = 이직 감지"
    contact_transitions }o--|| company_network_edges : "이직 흐름 가중치"
    contact_transitions }o--|| career_paths : "회사 시퀀스 누적"
    company_insights }o--|| company_reputation : "회사별 평판 집계"
    company_insights }o--|| salary_benchmarks : "직군+규모 연봉 집계"
    recommendation_scores }o--|| company_reputation : "추천 회사 평판"

    user_profiles {
        uuid id PK
        text current_company "내 현재 회사"
        text namecard_provider "remember 등"

    namecards {
        uuid id PK
        uuid owner_id FK
        text contact_hash "비식별 해시"
        text company_name "회사=네트워크 노드"

    }
    contact_transitions {
        uuid id PK
        text from_company "이전 회사"
        text to_company "이직한 회사"
        int observed_count "교차검증"
    }
    company_network_edges {
        uuid id PK
        text company_a "회사 A"
        text company_b "회사 B"
        int edge_weight "동시등장"

    }
    career_paths {
        uuid id PK
        text contact_hash "한 사람"
        text_array company_seq "회사 궤적"
        smallint total_moves "이직 횟수"
    }
    company_insights {
        uuid id PK
        text company_name "회사명"
        int actual_salary "실연봉"
        smallint nps_score "추천의향"
    }
    company_reputation {
        uuid id PK
        text company_name "회사명 UK"
        decimal avg_nps "평균 NPS"
        int insight_count "응답 수"
    }
    salary_benchmarks {
        uuid id PK
        text job_category "직군"
        int salary_p50 "중앙값"
    }
    recommendation_scores {
        uuid id PK
        uuid user_id FK
        text company_name "추천 회사"
        decimal survey_fit "서베이 0~1"
        decimal network_proximity "네트워크 0~1"
        decimal career_similarity "경로 0~1"
        decimal final_score "최종 점수"
    }

Tables

테이블 14개, 카드로 설명

색 = 어느 축에 속하는지. 각 카드는 "무슨 역할인지 + 대표 컬럼"을 보여줘요.

📋 company_insights

사용자가 입력하는 회사 내부 정보. 가장 중요한 데이터 자산.

job_category직군
actual_salary실연봉(만원)
resignation_reasons이직 이유
nps_score회사 추천의향 0~10
⭐ company_reputation

회사별 평판을 자동 집계 (트리거). 최소 3건 이상만 노출 → 익명성 보호.

company_name UK회사명
avg_nps평균 NPS
insight_count누적 응답 수
💰 salary_benchmarks

직군·규모·경력별 연봉 분포(25/50/75 퍼센타일).

job_category직군
salary_p50중앙값
sample_count표본 수
👤 user_profiles

명함첩을 연동한 사용자. 서베이는 익명이지만 연동은 식별이 필요.

id PK사용자 ID
current_company현재 회사
namecard_count보유 명함 수
📇 namecards

명함 한 장 = 내가 아는 사람 1명 = 네트워크 노드. 이름/연락처는 해시로만.

owner_id FK명함첩 주인
contact_hash연락처 해시
company_name그 사람 회사
🔄 contact_transitions

같은 사람 명함이 회사 A→B 로 바뀜 = 그 사람의 이직.

from_company이전 회사
to_company이직한 회사
observed_count몇 명함첩서 관측
🕸️ company_network_edges

회사 ↔ 회사 연결 강도. 같은 명함첩에 두 회사가 같이 있으면 가까운 사이.

company_a / b연결된 두 회사
edge_weight동시 등장 횟수
transition_weight이직 흐름 횟수
🌿 career_paths

한 사람이 거쳐온 회사 순서. 비슷한 궤적인 사람의 "다음 회사"를 추천에 사용.

company_seq회사 시퀀스(과거→현재)
total_moves이직 횟수
🏆 recommendation_scores

최종 추천 점수. 3개 축을 가중 합산해 회사별 점수로 저장.

survey_fit서베이 적합도 0~1
network_proximity네트워크 근접도 0~1
career_similarity경로 유사도 0~1
final_score가중 합산 (자동계산)
🔔 alert_subscriptions

채용공고 알림 신청 이메일.

email이메일
timeline이직 시점
⚡ analysis_cache

같은 입력엔 AI 재호출 없이 캐시 반환 → 비용 절감.

cache_key UK입력 해시
result_json분석 결과
Relationships

관계, 한 줄씩 풀어보기

화살표를 말로 바꾸면 이렇게 됩니다.

user_profiles → namecards사용자 1명이 명함 여러 장을 가짐 (1:N). 명함첩을 통째로 들고 있는 구조.
namecards → company_network_edges같은 명함첩에 든 회사들끼리 "동시 등장" 엣지를 자동으로 만들어요 (DB 트리거).
namecards → contact_transitions같은 연락처(contact_hash)의 회사가 바뀌면 "이직 이벤트"로 기록.
contact_transitions → career_paths그 사람의 이직들을 순서대로 모으면 커리어 궤적(회사 시퀀스)이 됨.
company_insights → company_reputation설문이 쌓일 때마다 회사별 평균 평판을 다시 계산 (트리거 자동 집계).
user_profiles → recommendation_scores사용자마다 회사별 최종 추천 점수를 저장 → 재현·AB 테스트·튜닝 가능.
Data Flow

데이터가 추천이 되기까지

두 축의 데이터가 어떻게 흘러서 최종 추천으로 합쳐지는지 흐름도예요.

flowchart TD
    subgraph A["축① 서베이"]
      U[사용자 입력] --> CI[(company_insights)]
      CI -->|트리거| CR[(company_reputation)]
      CI -->|집계| SB[(salary_benchmarks)]
    end
    subgraph B["축② 명함첩 (외부 연동)"]
      RM[리멤버 명함첩] --> NC[(namecards)]
      NC -->|동시등장 트리거| NE[(company_network_edges)]
      NC -->|회사 변경 감지| CT[(contact_transitions)]
      CT -->|이직 흐름| NE
      CT -->|궤적 누적| CP[(career_paths)]
    end
    CR -->|서베이 적합도| RS[(recommendation_scores)]
    NE -->|네트워크 근접도| RS
    CP -->|경로 유사도| RS
    RS --> OUT([추천 결과 화면])

Recommendation Score

최종 점수는 3개를 섞어서 만들어요

recommendation_scores 테이블의 final_score는 BEFORE INSERT 트리거가 자동으로 계산합니다.

final_score =
서베이적합도 × 0.5 + 네트워크근접도 × 0.25 + 경로유사도 × 0.25
50%
서베이 적합도
(직군·경력·스킬)
25%
네트워크 근접도
(명함첩 인맥)
25%
경로 유사도
(비슷한 사람의 선택)
💡 가중치는 바꿀 수 있어요. 각 행에 weight_survey/network/career 컬럼이 따로 있어서, 나중에 "네트워크를 더 중요하게" 같은 튜닝이나 A/B 테스트가 가능합니다.
